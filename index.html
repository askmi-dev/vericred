🛡️ Architectural Conformance & Interoperability Report

A comprehensive mapping outlining where the VeriCred implementation meets the RFC 9901 standard and what needs to be upgraded for full enterprise production.

✅ Standard Conformance
  • Compact SD-JWT-VC Format: Conforming to RFC 9901, with standard tilde (~) formatting.
  • Alphabetical Digest Ordering: Sorted _sd list prevents fingerprinting attacks.
  • Disclosure Hashing: Digests are computed over the US-ASCII bytes of the Base64URL-encoded 3-element disclosure arrays.
  • Holder Binding (cnf): Public key confirmation is embedded using cnf.jwk parameters.
  • ECDSA ES256 Signature: Mathematical trust leverages standard Web Crypto P-256 signatures.
🧪 Simulation Layers
  • Static Trust Directory: Registry verified locally in store.js instead of external federations.
  • In-Memory Nonce Cache: TTL checks run in local state, not a distributed gateway.
  • Local Miner Loop: Blockchain proof-of-work runs inside local browser Web Workers.
  • Off-Chain Storage: Portable credential exports are downloaded directly as Local JSON files.
🚀 Production Roadmap
  • Hardware HSMs: Replace Web Crypto in-memory keys with dedicated cloud HSMs/Enclaves.
  • National EUTLs: Integrate national trusted list directories with active root-CA chains.
  • Decentralized DIDs: Query remote DID resolvers (e.g. did:web, did:ion) over secure DNSSEC.
  • Signed Status Lists: Fetch CDN-cached RFC 8812 revocation lists instead of local registries.

⚖️ Standards Gap Review

1. JWS Compact SD-JWT-VC Support (RFC 9901)

VeriCred fully implements the standard compact, tilde-separated base64 disclosure structure wrapped in JSON Web Signatures (JWS) conforming to RFC 9901. Both legacy JSON envelopes and standard-conforming compact representations are supported and verified by the gateway.

2. Static Trust List vs Live EUTL List

Trust registration is simulated in-memory/in-store. A production system fetches and validates digital signatures over XML European Union Trusted Lists (EUTL) to map trust dynamically.

3. Static DIDs vs Distributed Resolution

VeriCred DIDs are resolved locally via statically seeded keys. Production environments resolve methods like did:ion or did:web dynamically via blockchain/HTTPS lookups.

4. Local blockchain vs Verifiable Data Registries

Our blockchain is a fast, local educational tool written in JavaScript. Real verifiable data registries rely on decentralized ledgers or cryptographically audited status lists (RFC 1035 or static HTTPS).
